## 1. Executive Summary

**Verdict: DIVERGENCE-CONFIRMED; kernel-bug localization pending C-1 session 1.**

The TQ decode path (`flash_attn_vec_tq` kernel plus its surrounding FWHT / packed-cache encode / prefill / dispatch pipeline in mlx-native at `a28783e`) produces outputs that violate the replay test's own declared bounds (`nrmse < 0.15`, `max_abs_diff < 1.0`) on every one of the 30 layers at some decode position in `{1, 5, 10, 15, 20, 25, 30}`. The per-layer max nrmse (max over 7 sampled positions, using the kernel test's own formula `sqrt(sum_sq_diff / sum_sq_ref)`) ranges from **0.673 to 1.281**, a **4.5× – 8.5×** exceedance of the 0.15 bound. The `max_abs_diff < 1.0` kernel-test gate is violated at **196 of 210** (layer, pos) cells for `sdpa_out`.

What this session **does** prove: the TQ decode path produces numerical divergence beyond the replay test's declared bounds on every layer at long-enough decode position. The 69-byte sourdough regression is real and measurable, not a ~~perception artefact~~ or representation-floor noise — it exceeds the kernel's own claimed tolerance.

What this session **does not** prove: that the defect lives inside the TQ SDPA kernel math itself. The initial draft of this report made that claim based on a smoking-gun cell (layer=0, pos=1, sdpa_out `max_abs_diff=0.844`) and an incorrect reading of the replay-test thresholds. Three corrections invalidate that framing:

1. **Corrected formula.** The diff script used `nrmse = sqrt(MSE) / (|dense|.mean() + 1e-12)`; the replay test uses `nrmse = sqrt(sum_sq_diff / sum_sq_ref)`. Under the correct formula, at (L=0, P=1) `nrmse = 0.1017`, which is **within** the 0.15 bound at that cell.
2. **Corrected thresholds.** The replay test asserts `max_abs_diff < 1.0` (not `< 1e-3`; the 1e-3 is the analyst's internal breach-trigger). At (L=0, P=1), `max_abs_diff = 0.844 < 1.0`, which is **within** the kernel test's own max_abs_diff gate at that cell. The "844× the gating threshold" framing in the initial draft conflated two different gates.
3. **Byte-identity scope.** The byte-identical `q_normed / k_normed / v_normed` dumps at (L=0, P=1) cover only the **current-token** Q/K/V. Decode step 1 attends over a **prefilled cache of 22 tokens** (seq_pos=22). On the TQ path, those 22 prefill tokens live in packed-KV buffers with per-position norms, indexed through a centroid table, read via a ring with mask/softcap parameters. None of that state was dumped or compared. Byte-identity of current-token tensors therefore does **not** rule out bugs in: packed-KV encode, FWHT direction/scale, per-position norm computation, centroid-table content, ring/mask/softcap parameters, buffer binding, or dispatch thread-group layout.

The divergence begins in-bounds at (L=0, P=1) and **grows** with position: at L=0 the `max_abs_diff` reaches `12.39` at some later position among the 7 sampled, and across the 30 layers the peak max_abs_diff is `17.93` (L=7). This growth pattern is consistent with at least four separate hypotheses, any of which could be the primary cause or a contributor:

- H1 (kernel math): a bug in the `flash_attn_vec_tq` Metal shader that is activated by a distribution of inputs not covered by the synthetic replay test.
- H2 (FWHT direction / scale): the Q rotation + O inverse-rotation on the TQ path differs from what the kernel expects.
- H3 (packed-cache encode / prefill): the packed KV buffers + norms stored at prefill don't faithfully encode the dense K/V — the representation-floor hypothesis resurfaces here, scoped to encode, not to general KV quantization.
- H4 (dispatch / buffer binding): an off-by-something in the Rust-side dispatch (offsets, thread-group dims, seq-pos bookkeeping) causes the kernel to read wrong or misaligned cache state.

The next session (ADR-007 Phase C-1) **must localize before editing**. Section 6 specifies the four-step localization matrix. Phase C-4 (fallback removal) remains deferred until C-1 both localizes and closes.

**Thresholds used in this report:**
- `max_abs_diff < 1.0` = **kernel test's own declared max_abs gate** (`test_flash_attn_vec_tq.rs:403`). Any `(L,P)` cell exceeding 1.0 is a confirmed bound violation.

- `max_abs_diff > 1e-3` = analyst's internal breach-trigger for surfacing small divergences. Retained for continuity with earlier sessions but **not** a kernel-level bound. The "844× threshold" language in the initial draft stemmed from a mis-identification of this internal trigger as the kernel gate.

## 4. Binary Verdict and Rationale

**VERDICT: DIVERGENCE-CONFIRMED; kernel-bug localization pending C-1 session 1.**

### Logic Chain (corrected)

Given:
1. On all 30 layers and at some decode position in `{1, 5, 10, 15, 20, 25, 30}`, `sdpa_out` exceeds both kernel-test bounds (`max_abs_diff < 1.0`, `nrmse < 0.15`); per-layer max nrmse range `[0.673, 1.281]`, per-layer max max_abs_diff range `[7.43, 17.93]`, 196 / 210 cells violate the 1.0 bound. [raw.csv + Table 3.3]
2. At the earliest measured cell (L=0, P=1): current-token Q/K/V are byte-identical (max_abs_diff = 0.0 for each); sdpa_out differs by 0.844 (within bounds but non-zero). [Table 3.4]
3. The prefilled-cache state consumed by the TQ kernel at decode step 1 (packed K, packed V, per-position norms, centroid table, ring indices, mask/softcap params for seq_pos 22) was not dumped on either branch during this session and therefore was not compared.

Therefore: the TQ decode path produces outputs that violate the replay test's declared bounds on every layer. The bug is somewhere in the chain `{encode → prefill → FWHT → kernel math → dispatch}` between the Q/K/V feeding the dense path and the dense path's sdpa_out, and the corresponding chain on the TQ side. The byte-identity of current-token Q/K/V at (L=0, P=1) rules out only the **current-token** input path as the source; it does **not** rule out the prefilled-cache path or the FWHT / encode / dispatch layers.

### What This Eliminates (narrowed from the initial draft)

| Candidate cause | Evidence | Eliminated? |
|-----------------|---------|-------------|
| Current-token K/V quantization precision | K/V max_abs_diff = 0.0 at (L=0, P=1) | Yes — for the current-token, at this cell. Does not eliminate prefill-cache quantization. |
| Current-token Q path (pre-FWHT) | Q max_abs_diff = 0.0 at (L=0, P=1) | Yes — for the current-token Q. Does not eliminate the FWHT applied during kernel dispatch or the inverse-FWHT on the output. |
| Residual-stream propagation from earlier layers at L=0 | Layer 0 is the first layer | Yes — L=0 has no prior-layer input to corrupt. |

### What This Does NOT Eliminate (surfaced by Codex review)

| Candidate cause | Why still open |
|-----------------|---------------|
| H1 — TQ SDPA kernel math (Metal shader) | Still possible; but replay-test with synthetic inputs passed, and the real-input replay was NOT run this session. |
| H2 — FWHT direction / scale on Q (pre-kernel) and O (post-kernel) | Not measured. The Q we dumped is pre-FWHT. Whether the FWHT applied inside the kernel dispatch matches the inverse-FWHT applied on O is unverified. |
| H3 — Packed-cache encode at prefill (representation) | Not measured. Packed K, packed V, per-position norms, centroid-table content at the 22 prefilled positions were never dumped on either branch. The 69-byte sourdough regression could be an encode-quality issue; the measurements we have do not distinguish this from a kernel-math bug. |
| H4 — Dispatch / buffer binding / ring / mask | Not measured. A buffer-binding off-by-something, wrong thread-group dims, or wrong ring/mask parameter would produce exactly the pattern we see (bounded at early decode with bit-identical current-token inputs, unbounded at later decode as more prefill is read). |
| H5 — Centroid-table content or index mapping | Per-head pattern at L=0/P=1 (heads 12/13, single-dim spike) is consistent with a mis-decoded centroid. Not measured directly. |

The initial draft of this report claimed to eliminate FWHT, ring chronology, and residual-stream propagation based on byte-identity of current-token tensors. That claim was wrong: decode step 1 reads 22 prefilled-cache tokens, whose state the TQ path encodes and reads differently. The initial draft's Table "What This Eliminates" over-claimed; the table above is the corrected version.

### What This Does NOT Claim

- It does **not** say commit `e9fd6fc` (ring chronology fix) was wrong. That fix addresses a real off-by-one bug. Still necessary.
- It does **not** say the representation-floor hypothesis is dead. ADR-007 Phase C-4b (representation floor) is no longer the leading candidate but is not eliminated, because prefill-cache state was never dumped or differenced.
- It does **not** say the 5-day-old replay-test claim (`nrmse < 0.15`, `max_abs_diff < 1.0`) was dishonest. The replay test used synthetic Gaussian Q/K/V. The C-0 audit shows those synthetic inputs did not cover the distribution that production Gemma-4 decode exercises. The claim was accurate for the test inputs; it was incomplete as a guarantee for production decode.
- It does **not** pinpoint a line of code or a single hypothesis. It confirms divergence and lists 5 open hypotheses for C-1 to resolve.

## 5. Kernel-Replay Re-Verification

**Status: DEFERRED — rescoped after Codex review.**

### Why the Initial "Load-Q/K/V-Into-Replay-Test" Plan Was Insufficient

The initial draft proposed: load the (L=0, P=1) Q/K/V dumps into `test_flash_attn_vec_tq.rs`, run, observe pass/fail. Codex review flagged a gap: this would replay only the **current-token** inputs, while the kernel at decode step 1 actually consumes 22 prefilled tokens from packed-KV buffers, per-position norms, a centroid table, and ring/mask parameters. Dumping current-token inputs alone and calling the kernel would not reproduce the production call at all — the kernel would attend over whatever prefill state the test fixture happens to synthesize.

### Rescoped C-1 Session-1 Task (see Section 6 for full matrix)

The first C-1 task is no longer "load Q/K/V into replay test and judge the kernel." It is:

1. **Dump the prefill-cache state** on both branches at seq_pos=22 for the sourdough prompt: packed K, packed V, per-position norms, centroid table, ring indices, mask/softcap params. Write an infra patch (parallel to the +7/+6-line dump patch) to dump this state. Revert the patch after the dump, same discipline as this session.
2. **Diff the prefill-cache state** between dense and TQ. If divergent: the bug is in encode or prefill, not the kernel. Subsequent steps.
3. **If prefill state matches, replay the kernel with real prefill state injected.** Write a new test in `/opt/mlx-native/tests/test_flash_attn_vec_tq.rs` that loads the real (Q, packed-K, packed-V, norms, centroid-table) and calls the kernel. If pass: the bug is in dispatch (buffer binding, thread-group dims, ring/mask). If fail: the bug is in the kernel math.
4. **FWHT sanity.** Apply the TQ path's forward-FWHT to the dumped Q and the inverse-FWHT to the dumped sdpa_out. Verify round-trip identity on both branches. A one-sided FWHT scale/direction error would produce the observed pattern.
