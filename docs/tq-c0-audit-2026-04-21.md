## 1. Executive Summary

**Verdict: BUG-FOUND**

The TQ SDPA kernel (`flash_attn_vec_tq` in mlx-native at a28783e) produces wrong output beginning at layer 0, decode position 1. `max_abs_diff = 8.441916e-01` (raw.csv row: layer=0, pos=1, op=sdpa_out), which is **844× the 1e-3 gating threshold**. The kernel's own declared bound (nrmse < 0.15 from `test_flash_attn_vec_tq.rs`) is violated on every one of the 30 layers, with nrmse ranging from 1.25 to 2.69.

The critical finding is that at (layer=0, pos=1), Q/K/V inputs are **byte-identical** between the dense and TQ paths (max_abs_diff = 0.0 for all three ops). The divergence originates inside the TQ SDPA computation, not in KV representation, FWHT, ring chronology, or propagation from an earlier layer. This ends the year-long ADR-007 debate: the 69-byte sourdough regression is not a representation floor — it is a functional bug in the kernel.

The next session executes **ADR-007 Phase C-1** (bug fix in the TQ SDPA kernel). Phase C-4 (fallback removal) is deferred until C-1 closes.

**Interpretation:** At decode position 1 (the very first decode step), layer 0 receives Q, K, and V that are provably identical bit-for-bit between the dense and TQ paths. The TQ SDPA kernel reads those identical inputs and produces an output that differs from the dense SDPA output by 0.844 in max element-wise absolute difference, which is 844× the 1e-3 gate. This measurement rules out the following as root causes:

- **KV representation / quantization precision** — K and V are byte-identical at this point; they have not yet been encoded by the TQ quantizer for the purposes of this comparison (pre-encode K/V are compared).
- **FWHT correctness** — Q is byte-identical; the Hadamard transform has not yet operated differentially.
- **Ring chronology** — decode position 1 means the ring buffer has only seen 1 token; the ring-start off-by-one bug fixed by commit `e9fd6fc` requires a full ring to manifest.
- **Residual stream propagation from earlier layers** — this is layer 0, the first layer in the network. There is no earlier layer whose error could propagate here.
- **Caller-side buffering / binding** — if the error were in buffer layout or dispatch arguments, it would manifest on Q/K/V too, but those are byte-identical.

The bug is located inside the TQ SDPA kernel's computation given these inputs.

## 4. Binary Verdict and Rationale

**VERDICT: BUG-FOUND**

### Logic Chain

Given:
1. At (layer=0, pos=1): `q_normed` max_abs_diff = 0.0 (byte-identical) [raw.csv row 2]
2. At (layer=0, pos=1): `k_normed` max_abs_diff = 0.0 (byte-identical) [raw.csv row 3]
3. At (layer=0, pos=1): `v_normed` max_abs_diff = 0.0 (byte-identical) [raw.csv row 4]
4. At (layer=0, pos=1): `sdpa_out` max_abs_diff = 8.441916e-01 [raw.csv row 5]

Therefore: The TQ SDPA kernel (`flash_attn_vec_tq` in mlx-native at a28783e) produces wrong output given byte-identical inputs. The bug is inside the kernel computation, not in any data supplied to it.

### What This Eliminates

| Candidate cause | Evidence | Eliminated? |
|-----------------|---------|-------------|
| KV quantization precision (representation floor) | K/V max_abs_diff = 0.0 at (L=0, P=1) | Yes |
| FWHT correctness | Q max_abs_diff = 0.0 at (L=0, P=1) | Yes |
| Ring chronology off-by-one (commit e9fd6fc) | decode pos=1, ring not yet wrapped | Yes |
| Residual stream propagation from earlier layers | Layer 0 is first layer; no prior state | Yes |
| Caller-side buffer binding / dispatch argument error | Would affect Q/K/V too, which are byte-identical | Yes |

### What This Does Not Claim

- It does **not** say commit `e9fd6fc` (ring chronology fix) was wrong. That fix addresses a real off-by-one bug. It was necessary but not sufficient.
- It does **not** say representation floor is impossible at long context. It is moot: the first-order functional bug dominates at decode position 1 before any quantization error can accumulate.
- It does **not** say the 5-day-old replay-test claim (`nrmse < 0.15`, `max_err < 1e-3`) was dishonest. The replay test used synthetic Q/K/V inputs. The C-0 audit shows those inputs did not cover the specific numerical path that fails on real Gemma-4 decode at (layer=0, pos=1). The claim was accurate for the test inputs; it was incomplete as a guarantee for production decode.
- It does **not** pinpoint the exact line of code. The per-head analysis (Section 3.5) narrows the suspect to the softmax or attention-weight accumulation path based on the peaked single-dim spike pattern in heads 12/13. The bug-fix session (C-1) will isolate further via kernel-replay unit testing with real inputs.
