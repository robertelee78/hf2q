## 1. Executive Summary

The two harness defects diagnosed in the C-1 VERIFICATION_BLOCKED report are fixed. Two teams
(Claude, Codex) independently patched `examples/tq_kernel_replay.rs`; both produced
byte-identical output on all four variations (`cmp` = 0 on `A`, `B`, `C`, `A_canary`). The
23-row TQ cache state was captured via a new default-off dump site at
`src/serve/forward_mlx.rs` (hf2q main `fdc0eb70`), gated on
`HF2Q_DUMP_TQ_STATE=1 && kv_seq_len==23 && layer_idx==0`; sha256 rows 0..21 gate passes on
both K and V.

Against the corrected harness, single-step L=0 / P=1 (kv_seq_len=23, ring_start=0,
mask_type=2, sliding_window=1024) shows:

| Variation | nrmse | max_abs_diff |
|---|---|---|
| **A** full TQ pipeline (FWHT → TQ SDPA → iFWHT, 3 barriers) | **5.14e-05** | 7.05e-04 |
| **B** FWHT disabled (1 barrier, no FWHT dispatches) | 1.2374 | 13.15 |
| **C** dense `flash_attn_vec` control on F32-dequantized K/V | **8.07e-05** | 1.65e-03 |
| **A_canary** in-range `k_norms[0,10] *= 2.0` mutation | 0.1111 | 1.54 |

**Reading:** A (5.1e-5) and C (8.1e-5) agree to within reduction-tree / NWG noise; B
(1.24) proves FWHT is doing necessary work; canary delta (0.111 ≫ 0.01) proves the
kernel reads in-range position 10. AC6 passes: A moved from C-1's pre-barrier 1.2445 to
5.1e-5 — a 24,000× swing that confirms barriers are gating dispatches.

**But scope is the point.** The CPU oracle uses the same `nibble_dequantize` as the kernel,
so the 4-bit quantization floor is intentionally cancelled on both A and C; the harness
measures "kernel implements its own spec at this captured state," not "kernel output
matches the end-to-end correct answer across the whole decode trajectory." H1/H2/H4 are
**not the bug at single-step L=0/P=1**; they remain open for multi-step, ring-wrap, and
nonzero-`ring_start` evolution.

### Verdict

- **H1 (kernel math)** — NOT the bug at single-step. Kernel matches its own CPU spec at
  kv_seq_len=23, ring_start=0. Harness cannot detect a spec-level dequant bug shared by
  both sides.
- **H2 (FWHT bracketing)** — NOT the bug at single-step. Disabling FWHT (B) diverges the
  output to nrmse=1.24; FWHT is doing real work. No evidence of wrong-direction at this
  state.
- **H4 (dispatch / mask / ring)** — NOT the bug at single-step. Packed-path dispatch, GQA
  head sharing, and the kv_seq_len=23 / ring_start=0 / sliding_window=1024 manifest work
  correctly. Ring-wrap and nonzero `ring_start` are explicitly out of scope for this
  harness.
- **Single-step clear; multi-step open.** Any remaining TQ defect now has to live in
  cache evolution, ring-wrap, cumulative drift, or full-graph concurrency scheduling —
  not in the single captured state this harness replays.

## 4. Known caveats (scope honesty)

### 4.1 Harness validates ONE captured state only

L=0, decode position 1, kv_seq_len=23, ring_start=0, mask_type=2, sliding_window=1024,
softcap=0. Single-step replay cannot expose cumulative drift, cache-evolution bugs, or
state corruption that appears after many decode steps. Ring-wrap (`kv_seq_len ≥
kv_capacity`) and nonzero `ring_start` are explicitly untested.

### 4.2 CPU oracle inverse-FWHTs the same packed bytes

`nibble_dequantize()` at `examples/tq_kernel_replay.rs:288–303` and `562–572` inverse-FWHTs
the packed K/V back to natural basis before `cpu_sdpa()`, so both A and C are scored
against the same dequantized logical cache and the 4-bit quantization floor is
intentionally cancelled. This is the right methodology for a single-step SDPA replay
harness, but it is **not** an end-to-end quantization-fidelity test. A defect in
`hadamard_quantize_kv` upstream of this captured state, or in cache production, is **not**
detectable here.

### 4.3 Symmetric canary-oracle mismatch (present in both implementations)

Both Claude's and Codex's implementations write the canary `×2` mutation into the GPU
`k_norms` buffer, but the CPU reference at the same call site uses the uncloned
`k_norms_ref` (unmutated baseline). The resulting 0.111 nrmse delta is well-defined —
it measures what the kernel produces on a perturbed norm vs the natural-basis reference
— but it is **not** the same quantity as "delta vs a CPU oracle given the mutated norm
identically." This conflates kernel-sensitivity-to-perturbation with
norm-vs-output-coupling. AC4's pass is preserved (the kernel provably reads position 10,
since a mutation there does move output by 0.111 ≫ 0.01), but the canary is a weaker
oracle than spec intended. Filing as **followup for next session**, not a merge blocker —
outputs are byte-identical between teams, so the conclusion is the same regardless of
which merges.

### 4.4 CPU reference scope

`cpu_sdpa()` in both implementations ignores `mask_type`, `sliding_window`, `ring_start`,
and `softcap`. This is fine for this manifest (`kv_seq_len=23 < sliding_window=1024,
ring_start=0, softcap=0`) but it is **not** a general replay oracle. Before reusing
this harness outside the current 23-row case, the CPU reference must be extended.

### 4.5 Variation C canary does not propagate to dense buffers

`--variation C --canary` would not perturb the F32 dense K/V (the mutation is applied to
`k_norms_padded` before it is reshaped into `k_norms_gpu`; the dense path builds from
`k_dequant` which is derived pre-mutation). Either propagate canary into dense buffers or
reject the flag combination. Filed as followup.

### 4.6 Raw `memory_barrier()` vs range-aware `barrier_between`

The harness uses bare `encoder.memory_barrier()` on a `CommandEncoder`; production uses
`GraphSession::barrier_between(&reads, &writes)`. At the Metal level both collapse to
`memoryBarrierWithScope:MTLBarrierScopeBuffers` and are functionally equivalent for this
3-dispatch case. Low severity.

## 5. Merge winner — Claude

Both teams produced byte-identical output on all four variations, so functional
correctness is equal. Claude's implementation (`fbb01d5` + `2d6d425`, +360 / −183 lines)
is selected over Codex's (`539adc6`, +229 / −235 lines) on four quality axes:

1. **Richer `ReplayMetrics` schema.** Canary typed as `String` (`"none" | "in-range" |
   "out-of-range"`) vs Codex's `bool`; preserves information in the JSON artifact for
   downstream analysis.
2. **Explicit `tmp_dense_buf`** for Variation C dense dispatch rather than
   `max(tq_tmp, dense_tmp)` sharing; cleaner separation.
3. **`serde` aliases on `Manifest` fields** for backward compatibility with older
   manifest layouts.
4. **Commit structure.** Two focused commits with accurate log messages; Codex's
   single-commit version carries two medium-severity stale comments at lines 491–495
   (incorrectly groups A+C for FWHT treatment) and 661–662 (claims write is deferred
   when `main()` enforces it) that survived both implementation and self-review.

Codex's implementation does add one strict improvement — a `ManifestShaGate` struct that
hard-exits if `rows_0_21_{k,v}_match = false`. Claude's version omits this. The
instrumenter already green-gates upstream, so it is not load-bearing; filed as a small
followup polish for hf2q main.

## 9. Meta-observation — dual-mode value

This is the third session in the TQ story (C-0 audit, C-0b localization, C-1 kernel
replay / VERIFICATION_BLOCKED). The "methodology error tier shifts per session" pattern
continues to hold: C-0 had a scope-matching error (dense vs TQ at different `kv_seq_len`;
retracted as false in C-1), C-0b shifted to a localization ambiguity, C-1 shifted to
two independent harness defects, and C-1 unlock shifted to scope-of-oracle caveats
(symmetric canary defect, CPU-reference masks assumption). Each session's scrutiny
surfaced a new class of error; the defect did not disappear, it relocated up the
oracle's stack.

Dual-mode earned its cost this session **via the read-only review path, not the
dual-harness path.** Codex's harness worker hung 2h 39m in a `memory-retrieve` poll
loop (0 JSONL bytes, 0 worktree commits; see `agents-harness-codex-result` / brain
memory `9a4cbc57` — `codex exec` in background `Bash` needs `< /dev/null` stdin
redirect). It was killed and rerun in read-only mode against Claude's harness. The
resulting independent review caught one med-severity canary defect Claude's own review
had not surfaced (symmetric, §4.3) and the "CPU oracle cancels the quant floor"
clarification that now anchors §4.2 of this report — a scope caveat that would otherwise
have gone unstated and left us claiming more than the harness supports.

So dual-mode's value this session was **adversarial cross-review, not parallel
implementation.** The `< /dev/null` pattern is captured as shared knowledge so future
sessions don't re-discover the hang. For the next multi-step session the recommendation
is: either pre-apply the stdin-fix in the cfa wrapper and re-enable dual-implementation,
or explicitly run solo + cross-review if the cost/benefit doesn't justify dual-impl for
that scope.
