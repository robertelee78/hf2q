## 2. TL;DR (≤300 words)

- **What oMLX is.** A FastAPI server that wraps Apple's Python MLX stack (mlx-lm + mlx-vlm + dflash-mlx) with a vLLM-style paged KV cache that spills to per-block safetensors files on SSD. It is a thin orchestration layer over Apple's stack — not a kernel-owning project. License: Apache 2.0; author: Jun Kim.
- **Does the SSD-KV-cache trick work as advertised.** The mechanism is real and mechanically sound: chain-hashed 256-token blocks, one safetensors file per block, atomic temp-file rename, async background writer (commit `af97a0f`, 2026-04-30 — same day as this research), startup directory scan rebuilds the index. The "milliseconds" framing is honest for hot-tier hits (sub-ms) and per-block SSD reads (~2 ms for a 5–10 MB block at 5 GB/s NVMe), but a 32K-token cold restore is **128 blocks × ~2 ms = ~256 ms of synchronous I/O on the inference thread**, not "instant."
- **Does it help agentic coding.** Real but narrow. The win is concentrated on (a) cold session resume and (c) parallel agents sharing a system prompt. Within a live session the KV is already in GPU RAM and persistence delivers nothing. Edit-in-middle (the most common agentic retry pattern) gets zero benefit because chain-hashing invalidates from the edit forward. The pitch's "every time an agent circles back" framing is misdirected for the typical loop shape.
- **Port to Rust: conditional yes — and the case is stronger than the source dossiers framed it.** hf2q already has axum serve mode, a single-slot `PromptCache`, **and an in-flight ADR-005 Phase 4 multi-model hot-swap stack** (`HotSwapManager<E>` at `serve/multi_model.rs:682`, ollama-style LRU pool, N=3 cached models, 80% memory ceiling). Under Ollama-style serving, KV-in-RAM evicts whenever the owning model evicts, which makes scenario (b) "mid-session eviction" + the new scenario (e) "model swap-out → swap-in on same context" the **dominant** wins, not the niche ones. The persistence layer is the natural sibling of the eviction signal that already exists. Top-3 first items: (1) falsification harness on M5 Max measuring restore-vs-prefill at 8K/32K, (2) hook the writer queue to `HotSwapManager` eviction so KV blocks spill to SSD when a model evicts, (3) full-attention only first; defer Qwen3.5 DeltaNet boundary snapshots until the dense path proves out.

## 3. What oMLX actually is

A Python orchestration layer over Apple's MLX stack. From `/opt/omlx/pyproject.toml` and module docstrings:

- Origin: vllm-mlx v0.1.0 evolved (carries vLLM's paged-KV vocabulary and design idioms).
- Author: Jun Kim (`junkim.dot@gmail.com`).
- License: Apache 2.0.
- Server: FastAPI + Uvicorn; admin panel benchmark tool described in `/opt/omlx/README.md:203-208`.
- Stack: depends on `mlx-lm` (Apple), `mlx-vlm` (Apple-adjacent), `dflash-mlx`. oMLX itself does not own any Metal kernels — `turboquant_kv.py` is explicitly a 6-line wrapper over `mlx_vlm.turboquant` (`/opt/omlx/omlx/turboquant_kv.py:2-7`).
- Distinct positioning vs hf2q: oMLX is a *server* orientation; hf2q is a *pure-Rust pipeline + CLI + small server* orientation with mlx-native kernels owned in-house.

The repo's own value-add is the persistence layer and continuous-batching scheduler, not the inference primitives.

## 4. The KV-cache mechanism

### 4.1 Block layout & shape

A "block" is a fixed contiguous span of tokens with metadata, ref count, hash, and optional cache data. `CacheBlock` carries `block_id`, `block_hash`, `token_count`, `ref_count`, and `cache_data` (`/opt/omlx/omlx/cache/paged_cache.py:126-162`). In paged-SSD-only mode `cache_data` is intentionally `None` and tensors live on disk (`/opt/omlx/omlx/cache/tiered_manager.py:8-12`).

Default block size: **256 tokens** on the scheduler path (`/opt/omlx/omlx/scheduler.py:321-331`). The factory's lower-level default is 64 (`/opt/omlx/omlx/cache/factory.py:41-45`); the scheduler can also align block size to RotatingKVCache window sizes (`/opt/omlx/omlx/scheduler.py:898-938`). Effective block size for any given model must be confirmed at runtime — it is not a global constant.

Per-layer tensor shape for a standard MLX `KVCache` is `(batch, n_kv_heads, seq_len, head_dim)`; block slicing is along axis 2 (`/opt/omlx/omlx/cache/type_handlers.py:164-170, 198-224`). dtype is preserved on disk (BF16/F16/F32 all flow through unchanged — `/opt/omlx/omlx/cache/paged_ssd_cache.py:155-194`).

Size math, two reference points:
- **oMLX block, BF16, ten full-attention layers, 256 tokens**: ~5.24 MB per block; 32K context = 128 blocks ≈ **670 MB on SSD** for full-attn KV alone (Claude's number, consistent with the layout).
- **hf2q block, F32, Qwen3.5-MoE full-attention layer**: `[n_seqs=2, n_kv_heads=1, max_seq_len=2*256, head_dim=256]` × 4 B → **~1 MiB per layer per 256-token block** at F32 (`/opt/hf2q/src/inference/models/qwen35/kv_cache.rs:14-16, 305-337`); ten such layers = 10 MiB per block; 8K prefix = 32 blocks ≈ **320 MiB at F32** or **160 MiB at BF16**, before any DeltaNet snapshot.

### 4.2 Hot tier (RAM)

The hot tier is **not** the live model's KV — that lives in mlx-lm's `BatchGenerator`. The hot tier is an `OrderedDict[bytes, Dict]` mapping block hash → raw-bytes serialized cache entry (`/opt/omlx/omlx/cache/paged_ssd_cache.py:643-648`, **VERIFIED IN-CODE**). Default `hot_cache_max_bytes=0` disables it. Entries store `tensors_raw` plus shape/dtype metadata extracted from MLX arrays (`/opt/omlx/omlx/cache/paged_ssd_cache.py:1198-1245`) — the comment explicitly notes that storing live MLX arrays here would accumulate Metal allocations and risk IOGPUMemory underflow.

Eviction: LRU via `OrderedDict.popitem(last=False)`, evicted entries enqueued to the SSD writer if not in hot-cache-only mode (`paged_ssd_cache.py:708-736`, **VERIFIED**). Reads also re-end the entry to keep LRU order honest (`paged_ssd_cache.py:782-788`).

Note for the port: this is a serialized-cache acceleration tier, not a live-KV allocator. Putting `MlxBuffer` handles in there in Rust would replicate the same lifecycle bug oMLX explicitly avoids.

### 4.3 Cold tier (SSD): on-disk format

- One file per block hash: `cache_dir/{0-f}/{sha256_hex}.safetensors` — single hex-character fanout (`paged_ssd_cache.py:823-847`).
- Manual safetensors writer on the background thread: 8-byte little-endian header length, JSON header, concatenated tensor bytes (`paged_ssd_cache.py:246-297`). Crucially, the writer **does not call `mx.save_safetensors`** — raw bytes are extracted on the inference thread post-`mx.eval`, then pure-Python file I/O happens off-thread without touching MLX/Metal.
- Atomic publication: temp file + `os.rename` (`paged_ssd_cache.py:989-1007`).
- Header metadata: `omlx_cache_format_version` (currently `"2"`, bumped 2026 to fix a RotatingKVCache zero-padding bug — `paged_ssd_cache.py:74-88`), `block_hash`, `token_count`, `num_layers`, `model_name`, `created_at`, `layer_cache_types` JSON, `layer_meta_states` JSON for stateful layers.
- No compression; no on-the-way-down requantization. TQ state, when active upstream, is stored as flattened NamedTuple fields (`paged_ssd_cache.py:1141-1156`).

### 4.4 Chain-hash identity scheme — VERIFIED

```python
# /opt/omlx/omlx/cache/paged_cache.py:78-119  (verbatim)
def compute_block_hash(parent_hash, token_ids, extra_keys=None, model_name=None):
    hasher = hashlib.sha256()
    if model_name: hasher.update(model_name.encode("utf-8"))
    hasher.update(parent_hash if parent_hash else b"omlx-root")
    hasher.update(bytes(str(tuple(token_ids)), "utf-8"))
    if extra_keys: hasher.update(bytes(str(extra_keys), "utf-8"))
    return BlockHash(hasher.digest())
```

SHA-256 over `model_name || (parent_hash | "omlx-root") || str(tuple(token_ids)) || str(extra_keys)`. Both source dossiers reported this correctly. The `str(tuple(...))` encoding is Python-specific — a Rust port that wants byte-exact interoperability must reproduce that exact string format. (Most likely the Rust port should *not* aim for byte interop and should use a cleaner fixed-endian encoding instead; see §9.3.)

Lookup is a flat `Dict[bytes, ...]` (`paged_ssd_cache.py:409-428`, `paged_cache.py:378-398`). No trie, no radix tree. Chain hashing makes the dictionary do the trie's work implicitly.

### 4.5 Read path: the "milliseconds" math

The exact restore call is verified at `/opt/omlx/omlx/cache/paged_ssd_cache.py:1490-1497`:

```python
# Load directly on the inference thread (Metal-safe).
# SSD read for a ~10MB block takes ~2ms @ 5GB/s — negligible.
# Previous executor-based approach caused deadlocks when
# mx.load() in a worker thread contested Metal GPU resources
# with the main inference thread.
arrays, file_metadata = mx.load(str(file_path), return_metadata=True)
```

Three observations from the actual code:

1. **Synchronous on the inference thread by design.** Not mmap, not async, not direct I/O. An earlier executor-based variant caused Metal resource deadlocks and was removed.
2. **`mx.load` is the standard mlx-lm safetensors loader** — header parse + tensor allocation + bytes-to-array. The 2 ms/block claim is a comment estimate; in practice it includes safetensors framing and `mx.array` construction beyond raw read time.
3. **Sequential.** `reconstruct_cache` walks blocks in order (`/opt/omlx/omlx/cache/prefix_cache.py:1276-1437`), stops at the first invalid/missing block, and concatenates along axis 2.

Honest math at 32K context (128 blocks, 5–10 MB each): ~256–640 ms of pure SSD-side latency, plus mlx-load framing, plus concat. Still ~2 orders of magnitude below full prefill recomputation, but emphatically not "instant." For exact full-prefix hits, the scheduler additionally falls back to full prefill on stateful caches because MLX-LM needs at least one residual token (`/opt/omlx/omlx/scheduler.py:2552-2568`).

### 4.6 Async write path — commit `af97a0f` VERIFIED

`git show af97a0f` (Author: jundot, 2026-04-30 11:39:38 +0900, same day as this dossier) message confirmed verbatim. Architecture:

1. Inference thread: `mx.synchronize` + boundary-snapshot merge + a single batched `mx.eval`.
2. Submit `_async_store_cache_worker()` to a single-worker `ThreadPoolExecutor("omlx-store-cache")` (`/opt/omlx/omlx/scheduler.py:567-572`).
3. Worker holds `_extracted_cache` reference so MLX arrays are not GC'd mid-copy (`/opt/omlx/omlx/scheduler.py:735-781`).
4. `batch_generator.remove(uid)` deferred until the worker drains, preventing slot reuse mid-write (`/opt/omlx/omlx/scheduler.py:783-832, 3998-4010`).

Same commit also moved SSD `unlink()` for evicted blocks into the writer queue via `('unlink', file_path)` sentinel tuples (`paged_ssd_cache.py:976-987, 1804-1823`), with inline-fallback when the queue is saturated.

Reported impact (commit message, gemma-4-31b-4bit, 32K prompt, max_tokens 128/256/512/1024/2048, R²=1.0):
- store_cache main-thread time: **3,482 ms → 127 ms (96% reduction)**
- per-token TG: 22.66 vs 23.0 baseline (-1.5%, essentially noise)
- fixed per-request overhead: 460–465 ms

Architectural significance: persistence cost was multi-second on real workloads before this commit. Anyone porting this idea must do the async-store split from day one or eat the same tail latency.

### 4.7 Recovery on restart

`PagedSSDCacheManager.__init__` calls `_scan_existing_files()` (`paged_ssd_cache.py:849-876`), which walks `{0-f}/*.safetensors`, runs `mx.load(..., return_metadata=True)` for each file (header-only read), validates `omlx_cache_format_version == "2"`, and registers valid blocks in the in-memory index. `_tmp.safetensors` partials are silently ignored (only fully-renamed files match the scan glob). No manifest, no journal — the directory itself is the source of truth.

A `kill -9` between `mx.eval` and the writer flush loses queued blocks. Clean shutdown's `close()` flushes hot-cache-only entries and joins the writer thread (`paged_ssd_cache.py:1998-2035`). `/opt/omlx/omlx/cache/recovery.py:54-92` is **not** durable restart recovery — it is in-process corruption recovery (clears batch generator state, GC).

For a 100 GB cache (~19,000 blocks at 5.24 MB), header-only scan ≈ 2 s — acceptable.

### 4.8 Prefix sharing & CoW semantics

Block-level dedup at the index level. `store_cache` increments ref count on hash collision rather than rewriting (`prefix_cache.py:454-466`). `fork_block_table` shares logical block tables across requests; `_cow_copy_block` only copies on append-to-shared-last-block — and in paged-SSD-only mode it explicitly skips data copy because data lives on SSD (`paged_cache.py:1168-1246`).

Practical implication: parallel agents with the same system prompt share blocks until the first divergent token. Divergence is full-block granularity; partial overlap inside a 256-token boundary is not exploited.

### 4.9 TurboQuant on stored KV

`turboquant_kv.py` is a thin wrapper around `mlx_vlm.turboquant` (`/opt/omlx/omlx/turboquant_kv.py:2-7`). When a layer's cache state is already TurboQuant (NamedTuple fields like `TurboQuantMSEState(norms, indices)`), the SSD writer detects the `'__turboquant_v2__'` marker and serializes fields individually (`paged_ssd_cache.py:1141-1156`). Restore reconstructs codecs deterministically from `(head_dim, bits, seed)` (`turboquant_kv.py:75-94`) — no need to store codebooks. The external prefill path explicitly does **not** apply TurboQuant during prefill due to merge/trim and precision concerns (`scheduler.py:1362-1369`); TQ runs in `BatchGenerator` during decode only.

The on-SSD format saves bytes proportional to TQ ratio, but persistence is not "automatically quantized" — it stores whatever representation upstream produces.

### 4.10 SSM/DeltaNet boundary snapshot store

`/opt/omlx/omlx/cache/boundary_snapshot_store.py` captures intermediate states at block boundaries during prefill for non-sliceable layers (`ArraysCache`, hybrid linear-attention state). Boundary snapshots are ephemeral: stored to SSD during prefill, loaded back for the final persistence merge, then deleted (`boundary_snapshot_store.py:61-80`). Reconstruction uses `_find_walk_back_truncation_point` to locate the last block where every required non-sliceable layer has real (non-placeholder) state and truncates there (`prefix_cache.py:1401-1437`). `_rotating_subclass.py` exists specifically because MLX `RotatingKVCache.size()` reports the configured maximum rather than the actual restored buffer length, which would otherwise leak zero-padding into attention reads.

This is the engineering-heaviest piece of the system and the one with the most subtle correctness traps. **Any Rust port targeting Qwen3.5's hybrid Gated-DeltaNet + Gated-Attention architecture must replicate this design or drop hybrid-model support to "experimental" until it does.**

## 5. Does it help agentic coding?

> **§5 reframing note (added 2026-04-30 post-publication).** The analysis below originally treated scenarios (a)/(c) as the dominant wins and (b) as a partial win. That framing assumed the model **stays loaded**. Under hf2q's ADR-005 Phase 4 multi-model hot-swap (Ollama-style: N=3 cached models, LRU eviction on 80% memory ceiling, see §8), the model owning a session's KV can evict at any time when a different model is requested. When that happens, in-RAM KV is gone with it. Scenario (b) "mid-session eviction" therefore upgrades to **DOMINANT**, and a new scenario (e) — "model swap-out → swap-in same model on same context" — becomes a **STRONG WIN** (full prefill skip on every cycle). The honest framing under hot-swap is: persistence is what makes the multi-model serve mode actually fast for repeat use of the same model on the same context. The §5 table at the bottom now includes (e) and an updated (b).

The pitch is directionally right but overgeneralized. "Every time an agent circles back to a previous prefix, the entire cache is recomputed" is only true for systems that submit each turn as a fresh request and have no prefix cache at all. A live model process handling a continuous request already owns the KV state in GPU RAM — persistence delivers nothing for *that* case. **But** under multi-model hot-swap, "circle back" gains a second meaning: "swap back to the model that owns this context after using a different model." That second meaning is precisely what oMLX's design solves, and it is what hf2q's Phase 4 hot-swap will need solved alongside it.

oMLX's actual win is at request boundaries. At request cleanup, the scheduler stores prompt+output for non-reasoning models or prompt-only for reasoning models (`scheduler.py:3684-3692`), and the next request with the same tokenized prefix restores blocks via `fetch_cache` + `reconstruct_cache` (`scheduler.py:2525-2592`).

Four scenarios, quantified with hf2q's measured prefill speeds where available:

**(a) Cold session resume — STRONG WIN.** Process killed (machine sleep, OOM, manual restart, deploy), user resumes the same context. Without persistent KV: 32K tokens at hf2q's measured Qwen3.5 prefill ≈ 300–450 tok/s (see `/opt/hf2q/docs/ADR-005-inference-server.md:5837` — 4096 tokens in 13.749 s = ~298 tok/s on the recorded path) means **70–110 s of recomputation**. With oMLX-style cache: ~256 ms of SSD I/O + injection ≈ 100–200× faster cold start. Even the pitch's optimistic 941 tok/s number gives ~34 s saved at 32K. Either way, the win is dominant.

**(b) Mid-session memory pressure — DOMINANT WIN under hot-swap.** Originally rated "partial." Reweighted: in ADR-005 Phase 4 (N=3 models, LRU at 80% RAM ceiling), the *common* eviction trigger is not memory pressure on a single live request — it is `HotSwapManager::insert()` admitting a different model and evicting the LRU one. When that fires, the evicted model's `HybridKvCache` is dropped; without persistence, every subsequent request to that model pays full prefill. With SSD persistence, the eviction handler enqueues block-aligned K/V to the writer; subsequent re-admission restores blocks at SSD speed. oMLX's tiered manager treats memory-pressure eviction as paged-only (`tiered_manager.py:150-162`); hf2q's `HybridKvCache` is allocated up-front at `max_seq_len` (`kv_cache.rs:152-158`). The hot-swap eviction signal is the right hook because it fires *between* requests on the evicted model — no mid-stream correctness hazard.

**(e) Model swap-out → swap-in same model on same context — STRONG WIN, NEW.** A coding agent uses the primary code model for tool calls, occasionally switches to a vision model to read a screenshot, then returns to the code model. With N=3 cached models and 80% memory ceiling on M5 Max 128 GB (~102 GB), a 35B-MOE coder + a 27B vision model + a small embedding model just barely fit; any time a fourth model is admitted, an LRU eviction fires. Without persistence: full 32K prefill on every swap-back-in (~70-110 s on Qwen3.5 at hf2q's measured prefill rate, ~16-22 s at q4_0). With persistence: ~256 ms SSD restore. Net: **2 orders of magnitude TTFT win on every swap-back-in cycle**. The frequency depends on the user's actual workflow — measure it in scenario-(e) telemetry before assuming it's the killer scenario, but if a coder triggers ≥1 swap-back-in per hour, persistence pays for itself in days.

**(c) Parallel agents sharing a system prompt — STRONG WIN.** Same model, tokenizer, chat template, and tool schemas → same first-N block hashes. Later agents hit blocks persisted by earlier agents with no client-side cache API. Saves `(N-1) × prefill_cost(system_prompt)`. At a 4 K system prompt across 4 agents on hf2q, this is 3 × 4 K / 300 tok/s ≈ **40 s aggregate prefill saved**. Requires a long-lived server.

**(d) Edit-in-middle — ZERO WIN.** Chain-hashing invalidates from the edit forward. If an agent edits message 7 of 12, blocks 0–6 are reusable; everything after is recomputed. Most agentic frameworks that retry, rephrase, or rewrite tool arguments fall into this bucket. The pitch's "every time an agent circles back" framing is **misdirected for this case** — true circle-back requires byte-identical prefix.

### The invariant the user should anchor on

> Within a live session **on a model that stays loaded**, hf2q's `HybridKvCache` already retains KV in GPU RAM and persistence delivers nothing. The win materializes precisely when (i) the process restarts, (ii) a different model evicts the loaded one (Phase 4 hot-swap), or (iii) parallel agents share a prefix. The first two are restart-shaped events; the third is a multiplexing event. Persistence is the layer that makes those events cheap.

If the marketing's within-single-session "every time an agent circles back" framing is what attracted attention, that interpretation is misleading. The accurate framing for hf2q's roadmap is: "oMLX's design solves the restart tax, the model-swap tax, and the parallel-agent shared-prefix tax. The first two are exactly what ADR-005 Phase 4 makes frequent. The third is what a long-lived serve mode makes frequent. All three are now in scope." It does not solve in-session tool-loop latency on a stable model, which is dominated by decode throughput regardless of caching.

| Scenario | Context | Without cache (hf2q today) | With SSD cache | Win |
|---|---|---|---|---|
| (a) Cold process resume | 8 K | ~14 s prefill | ~100 ms restore + last-token prefill | ~140× |
| (a) Cold process resume | 32 K | ~70–110 s prefill | ~256 ms restore | ~300–450× |
| (b) Hot-swap evicts model, then re-admits | 32 K | ~70–110 s prefill on swap-back-in | ~256 ms restore | ~300–450× per swap-cycle |
| (c) 4 agents, 4 K shared prompt | aggregate | ~54 s aggregate | ~14 s + 3 × restore ≈ 14 s | ~4× aggregate |
| (e) Same model swap-back-in same context | 32 K | full prefill | SSD restore | ~60–85× TTFT (q4_0) / ~300×+ (Qwen3.5) |
| (d) Edit-in-middle | mid-context | full prefill from edit | partial up to edit | partial |
| Live session, model stays loaded | any | KV in GPU RAM | same | **0×** |

## 6. Benchmark-claim audit

Searched `/opt/omlx` for `941`, `190`, `16.7`, `60.3`. Both researchers and this audit independently confirm: **none of those numbers exist in the repo**. The README points to an external URL `https://omlx.ai/benchmarks` (`/opt/omlx/README.md:203-208`).

What the in-tree benchmark code actually measures (`/opt/omlx/omlx/admin/benchmark.py`):
- Single-request PP/TG: `prompt_tokens / TTFT` and `completion_tokens / generation_duration` (`benchmark.py:134-165`). The single-test path warns if cached_tokens > 0 because the expected benchmark has no cache hits (`benchmark.py:220-224`).
- Batch PP/TG: `prompt_tokens × batch_size / prefill_wall_time`, `total_generated / generation_wall_time` (`benchmark.py:302-320`).
- Prompts inject a unique UUID specifically to **prevent SSD cache hits** during benchmarking (`benchmark.py:108-113`).

Therefore:
- **941 tok/s PP @ 8K Qwen3.5 on M3 Ultra 512 GB**: UNVERIFIED in repo.
- **190 tok/s @ 8× batch (3.36×)**: UNVERIFIED, and even if true, this is **aggregate throughput**, not per-request. Each request would see ~24 tok/s per stream — well below hf2q's per-request bar.
- **405B 16.7 → 60.3 tok/s at 8× batch**: UNVERIFIED, same aggregate-vs-per-request caveat.
- **Hardware delta**: M3 Ultra 60-core GPU at ~819 GB/s memory bandwidth vs M5 Max 40-core GPU at ~410 GB/s — roughly 2× memory bandwidth. Even valid M3 Ultra numbers do not transfer to M5 Max under hf2q's `project_speed_bar_full_matrix` invariant.

Read against hf2q's release-gate definition (≥1.00× llama.cpp on the same hardware, per-request), these numbers are not directly comparable. Continuous batching trades per-request latency for aggregate throughput; the 3.36× speedup is a server-throughput metric.

## 7. The TurboQuant collision

Both projects implement **the same paper**: Zandieh, Daliri, Hadian, Mirrokni — "TurboQuant: Online Vector Quantization with Near-optimal Distortion Rate" (arXiv:2504.19874, April 2025). hf2q's reference is at `/opt/hf2q/docs/ADR-007-turboquant-kv-cache.md:7`. The naming collision is incidental: both independently named their implementation after the algorithm.

| Axis | oMLX | hf2q |
|---|---|---|
| Implementation | `mlx_vlm.turboquant` (Python/Metal upstream); `omlx/turboquant_kv.py` is a 6-line re-export wrapper plus `BatchTurboQuantKVCache` for continuous batching | Native Rust + custom Metal shaders in `/opt/mlx-native/src/shaders/hadamard_quantize_kv_fast.metal` and `/opt/mlx-native/src/ops/hadamard_quantize_kv.rs` |
| Algorithm | Multiple state types: MSE, Prod, Polar, PolarProd, Split (`turboquant_kv.py:24-43`) | TurboQuantMSE with **Hadamard substitution** (Walsh-Hadamard Transform replaces dense random rotation, O(d log d) vs O(d²)) — paper-faithful with one explicit engineering deviation |
| State machine | Active by default in oMLX; KV state stored in NamedTuples; codecs rebuilt deterministically from `(head_dim, bits, seed)` | ADR-007 CLOSED 2026-04-24, **TQ-8-bit is hf2q's default decode path** as of merge `42e840d`; dense is opt-out via `HF2Q_USE_DENSE=1`. Lloyd-Max HB SDPA, ~85 tok/s, Gate A cosine 0.9998, Gate B argmax divergence 0.8% |
| Persistence | Stores quantized fields when active; reconstructs to standard KVCache via dequantize | No persistence today |

The hf2q implementation is the more sophisticated one. The Hadamard substitution is a deliberate complexity tradeoff in the SDPA decode path that oMLX does not pursue. The on-disk TQ format is **not byte-compatible** between the two — both serialize NamedTuple-style fields but field names, dtypes, and rotation conventions differ.

**oQ vs ADR-014 DWQ are unrelated.** oMLX's `oq.py` is a model **weight** quantization system (mixed-precision per-layer bit allocation combining GGUF K-quant strategy + Unsloth Dynamic 2.0 + bnb MSE clipping — `oq.py:2-7, 30-50, 70-92`). hf2q's ADR-014 DWQ is KL+MAE gradient distillation for streaming-convert weight quantization. Different algorithms, different goals. Don't conflate.

## 9. The Rust port — proposed design

### 9.1 Boundary

New workspace member crate, `kv-prefix-cache`, in the hf2q Cargo workspace at `/opt/hf2q/crates/kv-prefix-cache/` (matches the `project_pure_rust_crate_factory` direction). Public surface:

```rust
pub trait BlockStore  { fn get(&self, h: &BlockHash) -> Option<Arc<BlockBytes>>; ... }
pub struct DiskBlockStore   { ... }   // safetensors-on-disk
pub struct RamBlockStore    { ... }   // hot tier, raw bytes, LRU
pub struct TieredBlockStore<H, C> { ... }
pub struct BlockPrefixCache { ... }   // chain-hash index + lookup
```

Model-specific snapshot logic stays in `/opt/hf2q/src/inference/models/qwen35/kv_persist.rs`. Do not put this in `/opt/mlx-native` initially: mlx-native exposes buffer primitives; persistence policy + format + model fingerprints belong to hf2q. Promote to a sibling crate later if it proves backend-agnostic.

### 9.2 Data plane

- **Block format**: safetensors via the `safetensors` crate (Hugging Face official, Apache 2.0, already used at `/opt/mlx-native/src/weight.rs`). Same wire format as oMLX so blocks are validateable against oMLX recordings during testing — but **not** byte-identical (see 9.3 on hash encoding).
- **Hot tier**: `parking_lot::RwLock<lru::LruCache<BlockHash, Arc<Vec<u8>>>>`. Raw bytes only, never `MlxBuffer` handles (oMLX's lifecycle bug, learned).
- **Cold-tier writes**: dedicated thread + bounded `crossbeam_channel` / `flume` channel. **Not** `tokio::fs` for the writer — keep the write path off the axum runtime so a slow disk does not back-pressure HTTP handlers. Atomic temp file + `std::fs::rename`. Mirrors `af97a0f` architecture exactly: inference path does `mx.eval`-equivalent (a `MlxDevice::wait`) + bytes extract + enqueue, writer thread does file I/O and `unlink` ordered with later writes of the same hash.
- **Restore**: `std::fs::read` initially, dropped into a fresh `MlxBuffer` via `MlxDevice::alloc_buffer_with_bytes`. **Do not memmap the restored file as the live MTLBuffer** — mmap on the inference thread causes page faults during attention compute that stall the Metal queue (oMLX's removed-executor symptom). Consider `memmap2` only for header parsing; the tensor bytes get a real copy. After harness data lands, evaluate whether `memmap2` over the whole file with `madvise(WILLNEED)` beats the copy.
- **BF16 handling**: `half::bf16` has proper byte representation in Rust — no `view(uint16)` Python trick needed. Raw bytes are `slice::from_raw_parts(ptr as *const u8, n_bytes)`.

### 9.3 Identity scheme

Use SHA-256 with a **fixed-endian binary encoding**, not Python's `str(tuple(token_ids))`:

```rust
pub fn compute_block_hash(parent: Option<&BlockHash>, tokens: &[u32], model_fp: &str) -> BlockHash {
    let mut h = Sha256::new();
    h.update(model_fp.as_bytes());
    h.update(parent.unwrap_or(&ROOT_SEED));
    for &t in tokens { h.update(&t.to_le_bytes()); }
    h.finalize().into()
}
```

- Fixed-endian little-endian over `u32` is faster, language-neutral, and unambiguous.
- `model_fp` must be richer than oMLX's `model_name` string: it should include model id + quantization recipe + architecture revision + tokenizer/chat-template fingerprint. Otherwise a re-quant of the same checkpoint silently poisons cached blocks.
- This is **deliberately not byte-compatible with oMLX**. Tooling can validate against oMLX-format files at the safetensors level; the hash chain is hf2q's own. Trying to be hash-compatible with `str(tuple(...))` would be a permanent compatibility drag.

### 9.4 Prefix matching

Flat `HashMap<BlockHash, BlockMeta>`. The chain structure does the trie work. No `radix_trie` or similar dependency. Lookup is O(1) per block; O(N_blocks) for a full prefix; tokenization will dominate.

### 9.5 Integration with hf2q

The natural extension of the existing `PromptCache` **and** the existing `HotSwapManager`. Two integration points, both already wired into the call graph:

1. **Generalize `PromptCache` → `BlockPrefixCache`**. Today it is single-slot full-response replay (`engine.rs:663-738`). The new design keeps full-response replay as the fast path for full equality but adds chain-hash lookup for partial prefix hits keyed on `(model_fingerprint, parent_hash, token_ids[..BLOCK])`.
2. **Pre-prefill hook in the engine worker** (the same place that calls `PromptCache::lookup`): tokenize the new prompt, walk the chain forward, find the longest matching block-prefix, restore K/V into `HybridKvCache` slots, set `current_len` = restored token count, prefill from there.
3. **Post-generate hook**: on request completion (and on any boundary checkpoint), extract block-aligned K/V chunks from `HybridKvCache` and submit to the writer queue. Hold an `Arc` reference to the cache during serialization to mirror oMLX's `_extracted_cache` lifecycle.
4. **Hot-swap eviction + admission hooks (`HotSwapManager` integration).** Concrete API: inject `Arc<dyn KvSpiller<E>>` into `HotSwapManager` parallel to the existing `Arc<dyn ModelLoader<E>>` field (`multi_model.rs:682-690`). This preserves the manager's generic-over-`E` posture, mirrors the existing loader-injection pattern (no `Engine` trait widening, test fixtures inject a no-op spiller), and is configured at startup in the same scope where `DefaultModelLoader` is wired. Two trigger sites, both inside `HotSwapManager::load_or_get`:
   - **`pre_evict(handle, &Arc<LoadedEngine<E>>)`** — fired in the admission loop's eviction sequence and in `HotSwapManager::evict()`, **before** `engines.remove(&k)` drops the `Arc<LoadedEngine<E>>`. Spiller does the GPU-side `mx.eval` + raw-bytes extraction synchronously (must complete before the Arc drops; engine is gone after) but enqueues the actual safetensors write to a background writer thread (oMLX's pattern: extract on inference thread, write off-thread — `paged_ssd_cache.py:246-297`).
   - **`post_admit(repo, quant, &Arc<LoadedEngine<E>>)`** — fired in the miss path, **between** `loader.load()` returning a fresh `LoadedEngine<E>` and the manager publishing the Arc into `engines`. Spiller consults the on-disk index for known session prefixes and pre-warms `HybridKvCache` slots BEFORE the first user request arrives, eliminating the prefill-vs-restore race.

   Today's `HotSwapManager` exposes neither hook: `pool.insert()` returns `Vec<LoadedHandle>` consumed internally, `evict()` returns `u64` bytes freed. Adding the spiller field is ~50–100 LOC + tests (small Phase 4 surface amendment, lands as ADR-017's first deliverable). KV blocks are model-namespaced under `<cache_root>/models/<slug>/kv/...` so `pre_evict` can slug-namespace by `(repo_id, quant)` directly from the `LoadedHandle`. **This is the integration point that turns Phase 4 hot-swap from "fast model-load + slow re-prefill" into "fast model-load + fast restore".** Without it, hot-swap performs reasonably for first-touch but pays the prefill tax on every same-model swap-back-in (scenario (e) in §5).
5. **KV blocks are model-namespaced**: each model's blocks live under `<cache_root>/models/<slug>/kv/{0-f}/<sha256_hex>.safetensors` — distinct from `serve/cache.rs`'s weight cache root and indexed independently per model. The chain-hash includes a model fingerprint `(repo_id, quant, arch_rev, tokenizer_hash)` so re-quanting the same checkpoint or upgrading the chat template invalidates the namespace cleanly. Cross-model prefix sharing is **not** supported (different head_dims, layer counts, learned representations).
6. **Snapshot API on `HybridKvCache`** (new methods, model-agnostic where possible):
   - `snapshot_full_attn_block(layer_rank, start, len) -> &[u8]`
   - `restore_full_attn_block(layer_rank, start, bytes)`
   - `snapshot_linear_boundary(layer_rank, token_pos) -> Vec<u8>`
   - `restore_linear_boundary(layer_rank, token_pos, bytes)`
   - `set_current_len(per_seq: &[u32])` with validation against allocated `max_seq_len`

Full-attention slot shape `[head_dim, n_kv_heads, max_seq_len, n_seqs]` (`kv_cache.rs:14-16, 305-337`) is sliceable along axis 2 — block extraction is a strided copy per head. Decode write at `gpu_full_attn.rs:1676-1698` already updates `current_len`; the persistence path consumes that contract.

### 9.6 mlx-native integration

`MlxBuffer` is `StorageModeShared` — same physical pages visible to CPU and GPU. Restore is a CPU `memcpy` into a freshly allocated buffer; no explicit MTLBlit or staging upload is needed. The first implementation copies on the CPU before the next prefill/decode command buffer is submitted. If the harness shows CPU copy is the bottleneck, add a `metal_blit_block` op in mlx-native that takes a CPU-side `Vec<u8>` and a target `MlxBuffer` slice and queues a blit on a dedicated upload command buffer.

KV copy primitives already exist (`/opt/mlx-native/src/ops/kv_cache_copy.rs:8-9`) — extend, don't recreate.

### 9.7 DeltaNet / SSM boundary snapshot — the hardest piece

Qwen3.5-MoE is a 3:1 hybrid: Gated DeltaNet → Gated Attention. DeltaNet conv_state + recurrent state is **not sliceable per token** the way full-attn K/V is — it's a recurrent state evolved by the layer at each step (`/opt/hf2q/src/inference/models/qwen35/kv_cache.rs:339-392`, `forward_gpu.rs:2651-2673`).

Direct port of oMLX's design: capture conv_state + recurrent at every block boundary during prefill, store as the block's "linear_attn_state" payload, and on restore choose the latest valid boundary. This is exactly oMLX's `BoundarySnapshotSSDStore` pattern (`boundary_snapshot_store.py`) plus `_find_walk_back_truncation_point` (`prefix_cache.py:1401-1437`).

**Recommendation: ship dense full-attention persistence first; gate hybrid behind an explicit `HF2Q_KVPERSIST_HYBRID=1` flag during the boundary-snapshot work.** A full-attn-only cache is *useful but not correct* for Qwen3.5 — it would skip prefill of full-attn layers but force re-evaluation of DeltaNet from token 0, losing most of the advertised win. Honesty check: if hybrid correctness slips, the headline win (cold resume on Qwen3.5) does not exist. Do not ship a partial path while implying the full one.

### 9.8 Server-mode prerequisite — already met

hf2q already has `cmd_serve` + axum + `HotSwapManager` (§8). The cache layer extends the existing `LoadedModel`/`PromptCache` boundary inside the worker thread plus the eviction signal in `HotSwapManager::insert()`, no new daemon needed for v0. A separate Unix-socket daemon for cross-CLI sharing is a v2 concern at most.

## 10. Recommendation

**Conditional YES on ADR-017 = Persistent Block Prefix Cache for serve mode — and the case strengthens once Phase 4 hot-swap is in scope.**

Dominant condition: **Phase A0 falsification harness on M5 Max must land first**. Under hot-swap, the gate that matters most is scenario (e) — same model swap-back-in same context — clearing the ≥5× TTFT speedup bar at 32K. Scenario (a) cold-resume and (c) shared-prefix are secondary wins. If scenario (e) fails to clear, the whole investment is questionable; if it clears decisively (≥30× as the math suggests), this becomes a load-bearing complement to Phase 4 rather than an optional upgrade.

The mechanism is sound and well-understood; the engineering risk is concentrated in the DeltaNet boundary snapshot for Qwen3.5. Do not ship a dense-only Qwen3.5 cache that implies hybrid correctness — that would violate `feedback_never_ship_fallback_without_rootcause`. Either ship hybrid-correct in Phase C or hold ADR-017 at "Phase B closed; hybrid deferred" with a dated exit condition.

Three non-negotiables informed by hf2q's standing memory:
1. **Async writer from day one.** oMLX's `af97a0f` data shows persistence cost is multi-second on the inference path otherwise. Replicating that bug in Rust is unforced.
2. **Harness first, sub-1% decode regression second.** A decode regression here would silently break the q4_0-flat 1.0457× release-gate posture (`project_speed_bar_full_matrix`).
3. **Spill-on-evict cannot bottleneck swap-in.** The Phase 4 user-visible latency is "request to a non-resident model returns slowly because we had to evict something." If our spill-on-evict adds ~1-2 s to that latency, we've made the hot-swap worse, not better. The spill must complete in the background while the new model loads in parallel — measure both wall times in the harness.

What I would actively kill on:
- If Phase A0 shows mlx-native CPU→Metal-buffer copy on restore costs >50 ms per 5 MB block, the trick does not survive contact with M5 Max DRAM economics — hold.
- If scenario (e) speedup is <5× at 32K (i.e., SSD restore + warm-up costs eat most of the prefill saving) — drop it.
- If, after Phase B2, telemetry from hf2q's own usage shows <1 same-model swap-back-in event per developer-day, the win frequency is too low to justify maintenance. Descope to scenario (a)+(c) only or close ADR-017 unmerged.

Coordination note: Phase B2 is sequenced behind iter-209 AppState wiring. Either (i) hold ADR-017 until iter-209 lands and then sequence Phase B2 on top, or (ii) propose folding the eviction-hook design into iter-209's scope so the integration is born together. Option (ii) is cleaner if Phase 4 is still an open ADR; check Phase 4's owner.
