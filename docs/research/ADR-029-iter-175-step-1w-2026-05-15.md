# ADR-029 iter-175 Step 1w — H-F threads_per_tg geometry FALSIFIED

**Date**: 2026-05-15
**HEAD**: hf2q `7e3c37b4`, mlx-native `18aac38` → +1 test commit
**Iteration**: 27 of /loop autonomous

## Hypothesis (H-F)

Peer's `kernel_mul_mv_q6_K_f32` dispatches with `MTLSize(32, nsg=2, 1)` (see
`ggml-metal-ops.cpp:2249`).  hf2q's `kernel_mul_mv_q6_K_f32_nr2` dispatches with
`MTLSize(2, 32, 1)` (see `quantized_matmul_ggml.rs:680`).  Both produce 2
simdgroups of 32 threads, and the kernel uses only `tiisg` and `sgitg` (derived
from linear thread position) — so the layouts should be functionally
equivalent.  However, the Metal compiler/scheduler **might** emit different
memory-access patterns or instruction scheduling depending on which axis the
simdgroup boundary lands on.

**Testable claim**: swapping hf2q's `(2,32,1)` to peer's `(32,2,1)` will
change per-call kernel runtime by >1% at gemma4 Q_sliding decode shape
(m=1, n=4096, k=2816).

## Method

Standalone test `tests/iter175_h_f_threads_per_tg_geometry.rs`:

* Same shader compile path (xcrun -O3 precompiled .metallib — Step 1m default).
* Same FC bake (slots 700/701/702 = 1).
* Same buffers, same threadgroups grid `(1024, 1, 1)`.
* Vary ONLY `threads_per_tg`.
* 3 alt-paired cycles (order reversed each cycle), BATCH=32, WARMUP=20,
  MEASURE=80, 3s gap between arms within a cycle, 5s gap between cycles.

## Result

```
[H-F] aggregate (3 cycles, alt-paired):
  hf2q (2,32,1)  mean: 19.19us  samples: [18.62, 20.07, 18.89]
  peer (32,2,1)  mean: 19.27us  samples: [20.11, 19.13, 18.58]
  delta: (32,2,1) is -0.42% vs (2,32,1)
```

**Verdict: FALSIFIED (within noise — geometry doesn't matter).**

## Implications

1. **The threads_per_tg layout difference between hf2q and peer is NOT
   contributing to the decode gap.** Crossed off the lever list.

2. **The Metal compiler emits equivalent code for both layouts** when the
   kernel uses `tiisg` and `sgitg` (not `thread_position_in_threadgroup.x`
   or `.y` directly).  This is consistent with how Metal Shading Language
   semantics work: simdgroup membership is determined by linearized thread
   index, regardless of which axis the simdgroup boundary appears on.

3. **Chesterton's fence check**: hf2q's choice of `(2, 32, 1)` was deliberate
   — the legacy q6_K mv (NOT nr2) uses 1 simdgroup × 32 threads × 2 rows-per-SG,
   so x=2 was for the 2-row simdgroup partition.  When iter-308 added nr2, the
   layout was carried over.  No correctness or performance reason to flip;
   leaving (2, 32, 1) as-is.

## Why this matters anyway

This is the kind of hypothesis that LOOKS plausible from reading peer source,
but turns out to be a no-op when measured.  Future investigators who notice
the dispatch-geometry difference and propose porting it now have a recorded
falsification.

Test artifact retained at `/opt/mlx-native/tests/iter175_h_f_threads_per_tg_geometry.rs`
for rebench at HEAD if the question recurs.
